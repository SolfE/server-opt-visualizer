# Project Plan

## 프로젝트 개요

이 프로젝트는 예약/예매 플랫폼을 소재로 서버 병목과 성능 개선을 시각적으로 확인하는 모놀리스 실험 프로젝트입니다.
목표는 많은 기능을 만드는 것이 아니라, 병목이 잘 드러나는 최소 기능을 구현하고 개선 전/후를 비교할 수 있게 만드는 것입니다.

## 프로젝트 목표

이 프로젝트는 모놀리스 구조 안에서 다음을 명확하게 보여주는 것을 목표로 합니다.

- 어떤 요청에서 병목이 발생하는가
- 병목이 왜 발생하는가
- 어떤 방식으로 개선할 수 있는가
- 개선 전/후에 어떤 차이가 생기는가

## 프로젝트 방향

이 프로젝트는 일반적인 예약/예매 서비스를 완성하는 것이 목적이 아닙니다.
서버 실험에 필요한 최소 기능만 구현하고, 그 위에서 성능 이슈를 재현하고 비교하는 데 집중합니다.

즉, 기능 중심 프로젝트라기보다 다음에 가까운 구조를 지향합니다.

- 병목이 잘 드러나는 예제 서비스
- 최적화 실험용 모놀리스 애플리케이션
- 성능 지표와 구조 설명을 함께 보여주는 시각화 프로젝트

## MDD 방향

초기 구현 단계부터 모델링 문서화를 함께 관리합니다.
여기서 MDD는 복잡한 자동 생성 중심 접근보다, 도메인 구조와 흐름을 먼저 모델링하고 이를 문서와 코드에 일관되게 반영하는 실천에 가깝습니다.

우선 문서화할 대상은 아래와 같습니다.

- 핵심 도메인 모델과 관계
- 요청 흐름과 상태 변화
- 병목이 발생하는 지점과 원인
- 개선 전/후의 구조 차이

모델링 문서는 구현을 대체하는 것이 아니라, 기능 추가와 리팩토링의 기준선 역할을 해야 합니다.
도메인 복잡도가 커지면 초기 모놀리스 구조를 유지한 채 `reservation` 중심의 점진적 DDD 리팩토링도 검토합니다.

## 최소 기능 범위

초기 구현 범위는 아래 3가지로 제한합니다.

### 1. 공연 목록/상세 조회

- 공연 목록 조회
- 공연 상세 정보 조회

의도:
- 읽기 요청이 많이 몰리는 상황을 만든다
- 캐시 적용 전/후를 비교한다
- 조회 쿼리와 응답시간 변화를 확인한다

### 2. 좌석 조회

- 특정 공연 회차의 좌석 상태 조회

의도:
- 특정 데이터에 조회가 집중되는 상황을 만든다
- 상세 조회보다 더 무거운 읽기 요청을 만든다
- 좌석 데이터 조회 비용과 병목을 보여준다

### 3. 예매 요청

- 특정 좌석을 예매하는 요청

의도:
- 쓰기 요청의 병목을 만든다
- 동시성 문제와 트랜잭션 설계를 보여준다
- 단순 조회 최적화와 다른 접근이 필요함을 보여준다

## 병목 실험 포인트

이 프로젝트에서는 다음과 같은 병목을 우선적으로 다룹니다.

### 읽기 병목

- 공연 목록 조회 시 반복적인 DB 접근
- 공연 상세/좌석 조회 시 특정 데이터 집중 조회
- 캐시가 없을 때 응답시간 증가

### 쓰기 병목

- 예매 요청 시 동시성 충돌
- 좌석 중복 예매 방지 로직
- 트랜잭션 처리 비용

## 우선 실험할 개선 항목

### 1. 캐시 적용 전/후

- 공연 목록 조회
- 공연 상세 조회
- 좌석 조회

비교 포인트:
- 평균 응답시간
- 처리량
- DB 조회 감소 여부
- 캐시 히트율

### 2. 인덱스 적용 전/후

- 공연 조회 쿼리
- 좌석 조회 쿼리
- 예매 관련 조회 쿼리

비교 포인트:
- 쿼리 실행 시간
- 느린 쿼리 발생 여부
- 조회 응답시간 변화

### 3. 예매 요청 처리 개선 전/후

- 트랜잭션 범위 조정
- 락 또는 중복 방지 전략

비교 포인트:
- 성공률
- 충돌 발생 빈도
- 응답시간
- 처리 안정성

## 화면 구성 방향

화면도 많을 필요는 없습니다.
초기에는 최소한의 UI만 두고, 실험 결과를 보여주는 데 집중합니다.

### 메인 화면

- 프로젝트 소개
- 실험 대상 요청 소개
- 각 실험으로 이동할 수 있는 링크 또는 카드

### 실험 화면

- 요청 설명
- 병목 원인 설명
- 개선 방법 설명
- 개선 전/후 지표 비교
- 요청 흐름 또는 구조도

## 시스템 구성

초기 구조는 단순한 모놀리스로 시작합니다.

- Spring Boot App
- PostgreSQL
- Redis

이 구성만으로도 다음을 충분히 보여줄 수 있습니다.

- 캐시 적용 전/후
- 인덱스 적용 전/후
- 읽기 요청과 쓰기 요청의 차이
- 동시성 처리 이슈

## 확장 실험 후보

초기 구현은 PostgreSQL, Redis, 단일 모놀리스 구성에 집중합니다.
다만 실험 범위가 확장되면 아래 도구도 후속 과제로 도입을 검토합니다.

### Kafka

- 비동기 이벤트 흐름 실험
- 예매 후속 처리 분리 가능성 검토
- 동기 요청 중심 구조와 이벤트 기반 처리의 차이 비교

현재 단계에서는 필수가 아니며, 핵심 3개 흐름과 기본 병목 재현이 안정화된 뒤 도입 여부를 판단합니다.

### Grafana

- 성능 지표 시각화 강화
- 응답시간, 처리량, 에러율, 캐시 히트율 대시보드 구성
- 개선 전/후 비교 결과를 더 명확하게 표현

현재 단계에서는 간단한 서버 렌더링 기반 비교 화면을 우선하고, 계측 지표가 충분히 쌓이면 Grafana 연동을 검토합니다.

## 측정할 지표

초기에는 아래 지표만 명확하게 보여주면 충분합니다.

- 평균 응답시간
- 최대 응답시간
- 처리량
- 에러율
- 캐시 히트율
- 느린 쿼리 여부

## 제외할 기능

초기 단계에서는 아래 기능을 구현하지 않습니다.

- 회원가입/로그인
- 결제
- 알림
- 마이페이지
- 관리자 기능
- 추천/리뷰/즐겨찾기
- 복잡한 검색 필터
- 운영용 분산 시스템 구성

이 기능들은 서버 병목 실험의 핵심이 아니기 때문에 우선순위에서 제외합니다.

## 기술 방향

- Java 21
- Spring Boot
- Spring MVC
- Thymeleaf
- Gradle
- PostgreSQL
- Redis

후속 확장 후보:

- Kafka
- Grafana

초기에는 서버 렌더링과 단순한 시각화 방식으로 시작하고, 필요할 때만 차트나 메트릭 표현을 추가합니다.

## 개발 원칙

- 기능 수보다 병목 재현 가능성을 우선한다
- 실험 가능한 최소 기능만 구현한다
- 개선은 반드시 측정 가능한 기준과 함께 다룬다
- 복잡한 구조보다 설명 가능한 구조를 우선한다
- 모델링 문서를 코드와 함께 갱신한다
- 개선 전/후 비교가 화면에서 바로 보이게 만든다

## 초기 개발 우선순위

1. 모놀리스 실행 환경과 문서 구조 정리
2. 핵심 도메인 모델 초안과 흐름 문서화
3. 공연 목록/상세 조회 구현
4. 좌석 조회 구현
5. 예매 요청 구현
6. 병목 재현 시나리오 정의
7. 캐시 적용 전/후 비교
8. 인덱스 적용 전/후 비교
9. 예매 요청 동시성 처리 비교
