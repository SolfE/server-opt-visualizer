# Server Opt Visualizer

예약/예매 플랫폼을 소재로 서버 병목과 성능 개선을 시각적으로 확인하는 프로젝트입니다.
핵심은 많은 기능을 만드는 것이 아니라, 병목이 잘 드러나는 최소 기능만 구현하고 개선 전/후를 비교할 수 있게 만드는 것입니다.

## 프로젝트 목표

이 프로젝트는 모놀리스 구조 안에서 다음을 명확하게 보여주는 것을 목표로 합니다.

- 어떤 요청에서 병목이 발생하는가
- 병목이 왜 발생하는가
- 어떤 방식으로 개선할 수 있는가
- 개선 전/후에 어떤 차이가 생기는가

## 프로젝트 방향

이 프로젝트는 일반적인 예약/예매 서비스를 완성하는 것이 목적이 아닙니다.
서버 실험에 필요한 최소 기능만 구현하고, 그 위에서 성능 이슈를 재현하고 비교하는 데 집중합니다.

즉, 기능 중심 프로젝트라기보다 다음에 가까운 구조를 지향합니다.

- 병목이 잘 드러나는 예제 서비스
- 최적화 실험용 모놀리스 애플리케이션
- 성능 지표와 구조 설명을 함께 보여주는 시각화 프로젝트

## 최소 기능 범위

초기 구현 범위는 아래 3가지로 제한합니다.

### 1. 공연 목록/상세 조회

- 공연 목록 조회
- 공연 상세 정보 조회

의도:
- 읽기 요청이 많이 몰리는 상황을 만든다
- 캐시 적용 전/후를 비교한다
- 조회 쿼리와 응답시간 변화를 확인한다

### 2. 좌석 조회

- 특정 공연 회차의 좌석 상태 조회

의도:
- 특정 데이터에 조회가 집중되는 상황을 만든다
- 상세 조회보다 더 무거운 읽기 요청을 만든다
- 좌석 데이터 조회 비용과 병목을 보여준다

### 3. 예매 요청

- 특정 좌석을 예매하는 요청

의도:
- 쓰기 요청의 병목을 만든다
- 동시성 문제와 트랜잭션 설계를 보여준다
- 단순 조회 최적화와 다른 접근이 필요함을 보여준다

## 병목 실험 포인트

이 프로젝트에서는 다음과 같은 병목을 우선적으로 다룹니다.

### 읽기 병목

- 공연 목록 조회 시 반복적인 DB 접근
- 공연 상세/좌석 조회 시 특정 데이터 집중 조회
- 캐시가 없을 때 응답시간 증가

### 쓰기 병목

- 예매 요청 시 동시성 충돌
- 좌석 중복 예매 방지 로직
- 트랜잭션 처리 비용

## 우선 실험할 개선 항목

### 1. 캐시 적용 전/후

- 공연 목록 조회
- 공연 상세 조회
- 좌석 조회

비교 포인트:
- 평균 응답시간
- 처리량
- DB 조회 감소 여부
- 캐시 히트율

### 2. 인덱스 적용 전/후

- 공연 조회 쿼리
- 좌석 조회 쿼리
- 예매 관련 조회 쿼리

비교 포인트:
- 쿼리 실행 시간
- 느린 쿼리 발생 여부
- 조회 응답시간 변화

### 3. 예매 요청 처리 개선 전/후

- 트랜잭션 범위 조정
- 락 또는 중복 방지 전략

비교 포인트:
- 성공률
- 충돌 발생 빈도
- 응답시간
- 처리 안정성

## 제외할 기능

초기 단계에서는 아래 기능을 구현하지 않습니다.

- 회원가입/로그인
- 결제
- 알림
- 마이페이지
- 관리자 기능
- 추천/리뷰/즐겨찾기
- 복잡한 검색 필터
- 운영용 분산 시스템 구성

이 기능들은 서버 병목 실험의 핵심이 아니기 때문에 우선순위에서 제외합니다.

- 기능 수보다 병목 재현 가능성을 우선한다

- 개선은 반드시 측정 가능한 기준과 함께 다룬다
- 복잡한 구조보다 설명 가능한 구조를 우선한다
- 개선 전/후 비교가 화면에서 바로 보이게 만든다

## 초기 개발 우선순위

1. 공연 목록/상세 조회 구현
2. 좌석 조회 구현
3. 예매 요청 구현
4. 병목 재현 시나리오 정의
5. 캐시 적용 전/후 비교
6. 인덱스 적용 전/후 비교
7. 예매 요청 동시성 처리 비교

## 협업 규칙

### 커밋

- 한 커밋에는 하나의 의도만 담는다
- 커밋 메시지는 `prefix: summary` 형식으로 작성한다
- 사용 프리픽스: `feat`, `fix`, `perf`, `refactor`, `docs`, `test`, `chore`, `style`
- 예시: `feat: add seat lookup API`, `perf: optimize performance query`

### PR

- PR은 가능한 작게 나눈다
- PR 제목도 커밋과 같은 프리픽스를 사용한다
- 무엇을 바꿨는지와 왜 바꿨는지를 함께 적는다
- 성능 관련 변경은 전/후 차이를 같이 남긴다
- 예시: `perf: improve concert detail response time`

### 코드 스타일

- Java 코드는 Google Java Style을 기준으로 맞춘다
- 일관된 네이밍과 계층 분리를 유지한다
- 불필요한 추상화보다 읽기 쉬운 구조를 우선한다
- 복잡한 로직은 짧은 주석으로 의도를 남긴다

### 브랜치 전략

- `main`은 항상 실행 가능한 상태로 유지한다
- GitHub Flow를 사용한다
- 작업은 `feature/*`, `fix/*`, `perf/*`, `docs/*` 브랜치에서 진행한다
- 작업 완료 후 PR 리뷰를 거쳐 `main`에 반영한다

## 실행 방법

./gradlew bootRun
